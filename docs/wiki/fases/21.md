## 21a — Rate Limiting + CORS

**Lib:** `slowapi` (wrapper de `limits` para FastAPI)

### Rate limits implementados

| Endpoint | Limite | Key |
|----------|--------|-----|
| `POST /auth/login` | 5/min | IP |
| `POST /auth/forgot-password` | 3/hour | IP |
| `POST /auth/reset-password` | 3/hour | IP |
| `POST /sys-mgmt/login` (admin) | 5/min | IP |
| `POST /chat` | 20/min | `tenant:{tenant_id}` |
| `POST /api/whatsapp/webhook` | 100/min | IP |
| `POST /api/telegram/webhook/{token}` | 100/min | IP |

**Singleton:** `rate_limit.py` — `Limiter(key_func=get_remote_address, storage_uri=REDIS_URL or "memory://")`

**`get_tenant_key`:** extrai `tenant_id` do JWT **sem verificar assinatura** (só para determinar bucket de rate limit — a autenticação real ainda valida a assinatura).

**CORS:** `CORSMiddleware` com `allow_origins=[FRONTEND_URL]`, restrito no `api.py`.

## 21b — Criptografia em Repouso (Fernet)

**`src/docagent/crypto.py`:**
- `EncryptedString(TypeDecorator)`: cifra na escrita, decifra na leitura transparentemente
- Usa Fernet (AES-128-CBC + HMAC-SHA256) via `ENCRYPTION_KEY` env var
- Graceful degradation: se `ENCRYPTION_KEY` vazia, salva plaintext com warning

**Campos protegidos:**

| Tabela | Campo |
|--------|-------|
| `telegram_instancia` | `bot_token` |
| `tenant` | `llm_api_key` |
| `audio_config` | `elevenlabs_api_key` |
| `admin` | `totp_secret` |

**Gotcha crítico:** `bot_token` em `TelegramInstancia` tem `unique=True` mas `EncryptedString` é não-determinístico (IV aleatório por chamada). Em produção com `ENCRYPTION_KEY` ativo, queries `WHERE bot_token = ?` falham. Em dev/test sem key, funciona (plaintext). Fix futuro: coluna SHA-256 separada para lookup.

**Alembic:** `n4o5p6q7r8s9_encrypt_sensitive_fields.py` — resize de colunas para acomodar ciphertext.
**Script de migração:** `alembic/scripts/reencrypt_secrets.py` — idempotente, verifica se já está cifrado.

## 21d — 2FA TOTP para Admin

**`src/docagent/auth/totp.py`:**
- `gerar_secret()` → `pyotp.random_base32()`
- `gerar_qr_uri(secret, username, issuer)` → URI otpauth://
- `verificar_codigo(secret, codigo)` → `valid_window=1` (30s de tolerância)

**Fluxo de login com 2FA:**
```
POST /sys-mgmt/login
  → se totp_habilitado=False: retorna JWT normal
  → se totp_habilitado=True: retorna LoginResponse{requires_2fa: true, temp_token: JWT 5min}

POST /sys-mgmt/login/2fa {temp_token, codigo}
  → verifica temp_token (type=temp_2fa)
  → verifica código TOTP
  → retorna JWT definitivo
```

**`create_temp_token(admin_id)`:** JWT com `type=temp_2fa`, expira em 5min.

**Endpoints de gerenciamento:**
- `GET /2fa/setup` → gera secret, salva no banco, retorna `{qr_uri, secret}`
- `POST /2fa/confirmar` → valida código, ativa `totp_habilitado=True`
- `DELETE /2fa/desativar` → zera secret e flag

**Alembic:** `p6q7r8s9t0u1_add_totp_to_admin.py`

## 21e — Validação de Origem de Webhooks

### WhatsApp — `_validar_webhook_evolution(request)`
- Verifica header `apikey` vs `Settings().EVOLUTION_API_KEY`
- Se `EVOLUTION_API_KEY` vazio: aceita tudo (facilita dev local)
- Se preenchido e diferente: `HTTP 401`

### Telegram — `_validar_webhook_telegram(request, webhook_secret)`
- Verifica header `X-Telegram-Bot-Api-Secret-Token` vs `instancia.webhook_secret`
- Se `webhook_secret` não configurado: aceita tudo (backward compatible)
- Secret gerado em `TelegramService.configurar_webhook()` via `secrets.token_hex(32)`
- Registrado no Telegram via `setWebhook(url, secret_token=...)`

**Alembic:** `q7r8s9t0u1v2_add_telegram_webhook_secret.py`

## Pentest Interno — Vulnerabilidades Corrigidas

Pentest realizado com análise estática do codebase. 4 vulnerabilidades reais encontradas e corrigidas:

### 🔴 CRÍTICA — `tenant/router.py`: CRUD público sem autenticação
- **Exploit:** `curl -X DELETE http://api/api/tenants/1` (sem token)
- **Fix:** Endpoints removidos. CRUD já existia protegido em `/api/admin/tenants/*`

### 🔴 CRÍTICA — `usuario/router.py`: GET e PUT sem autenticação
- **Exploit:** `curl http://api/api/usuarios/5` lê dados de qualquer usuário sem token
- **Fix:** Adicionado `CurrentUser`/`CurrentOwner` + `usuario.tenant_id == current_user.tenant_id`

### 🟠 HIGH — `agente/documento_service.py`: IDOR em DELETE de documentos
- **Exploit:** Usuário do tenant1 pode deletar `doc_id` de outro tenant sabendo o ID
- **Fix:** `delete(doc_id, agente_id)` — verifica `doc.agente_id == agente_id` antes de deletar

### 🟠 HIGH — `atendimento/router.py`: Path traversal no media endpoint
- **Exploit:** `media_ref = "local:../../etc/passwd"` → arquivo servido
- **Fix:** `Path.resolve()` + `is_relative_to(audio_dir)` — bloqueia qualquer `..`

## Fixes de Áudio (correções durante a fase)

### Reprodução de áudio no painel de atendimento
**Problema:** `<audio :src="/api/atendimentos/media/{id}">` — browser não envia `Authorization` header → 401.
**Fix:** `fetch()` com token + `URL.createObjectURL(blob)` → `audioBlobUrls[msg.id]` como `src`.

### SSE com ID fake
**Problema:** SSE handler usava `id: Date.now()` como ID de mensagem.
**Fix:** Backend inclui `mensagem_id` real do banco em todos os payloads `NOVA_MENSAGEM`.

### Qualidade de transcrição (STT)
**Problema:** Modelo padrão `"base"` (74M params) tem qualidade ruim para português.
**Fix:**
- `AUDIO_STT_MODELO` default: `"base"` → `"small"` (244M params)
- `condition_on_previous_text=False` (evita alucinações)
- `initial_prompt="Transcrição em português brasileiro:"`
- `no_speech_threshold=0.6`
