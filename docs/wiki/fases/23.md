## Objetivo

Eliminar três fontes de estado em memória local que quebram silenciosamente com múltiplos workers:

| Componente | Problema | Solução |
|------------|----------|---------|
| `SessionManager` | dict local por processo → sessões perdidas entre workers | `RedisSessionManager` com pickle + TTL |
| `AtendimentoSseManager` / `Lista` | queues locais → worker B não recebe eventos do worker A | Redis Pub/Sub bridge pattern |
| `_agent_cache` nos routers | dict global que cresce sem limite, local por worker | `TTLCache(maxsize=100, ttl=1800)` |

Além disso, mover a ingestão de PDF para fora do request path via Celery.

## 23a — Redis

### RedisSessionManager (`chat/session.py`)

```python
class RedisSessionManager:
    def __init__(self, redis, ttl: int = 3600): ...
    async def get_async(self, session_id: str) -> dict:
        data = await self._r.get(f"session:{session_id}")
        return pickle.loads(data) if data else {"messages": [], "summary": ""}
    async def update_async(self, session_id: str, state: dict) -> None:
        await self._r.setex(f"session:{session_id}", self._ttl, pickle.dumps(state))
    async def delete_async(self, session_id: str) -> bool:
        return bool(await self._r.delete(f"session:{session_id}"))
```

`InMemorySessionManager` mantido como fallback (dev sem Redis, testes).
`SessionManager` é alias para `InMemorySessionManager`.
Factory: `get_session_manager_instance(redis=None)` retorna Redis ou in-memory conforme REDIS_URL.

### Interface async obrigatória

A migração para Redis async forçou a interface de sessão a ser 100% async:
- `ChatService.stream()` → `ChatService.astream()` (async generator)
- `ChatService.delete_session()` → `ChatService.delete_session_async()`
- Todos os callers nos routers (chat, telegram, whatsapp) trocados para `await sessions.get_async()` / `await sessions.update_async()`

### SSE com Redis Pub/Sub (`atendimento/sse.py`)

Bridge pattern: `broadcast()` publica no Redis; cada subscriber tem uma `asyncio.Task` que repassa mensagens do canal para sua `asyncio.Queue` local.

```python
async def broadcast(self, atendimento_id: int, event: dict) -> None:
    if self._redis is not None:
        await self._redis.publish(f"sse:atendimento:{atendimento_id}", json.dumps(event))
    else:
        for q in list(self._queues[atendimento_id]):
            await q.put(event)
```

Sem duplicação de mensagens: com Redis, o evento não vai direto para as queues locais — só via bridge.

### TTLCache nos routers

Em `chat/router.py`, `telegram/router.py` e `whatsapp/router.py`:

```python
from cachetools import TTLCache
_agent_cache: TTLCache = TTLCache(maxsize=100, ttl=1800)
_cache_lock = asyncio.Lock()
```

`_get_or_build_agent` virou `async def` com `async with _cache_lock`.

### Lifespan em `api.py`

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    redis_client = get_redis_client()
    if redis_client is not None:
        atendimento_sse_manager._redis = redis_client
        atendimento_lista_sse_manager._redis = redis_client
        _deps._session_manager = RedisSessionManager(redis_client)
    yield
    if redis_client:
        await redis_client.aclose()
```

## 23b — Celery

### `celery_app.py`

```python
celery = Celery("docagent",
    broker=f"{redis_base}/1",
    backend=f"{redis_base}/2",
    include=["docagent.tasks.ingestao"]
)
```

Broker em Redis DB1, backend em DB2 (separado do DB0 usado por sessões).

### `tasks/ingestao.py`

```python
@celery.task(bind=True, max_retries=3, default_retry_delay=30)
def ingerir_documento_task(self, agente_id: int, filename: str, content_b64: str) -> dict:
    content = base64.b64decode(content_b64)
    asyncio.run(_ingerir(agente_id, filename, content))
    return {"status": "ok", ...}
```

Usa `asyncio.run()` porque tarefas Celery são síncronas mas o ingest é async.

### Endpoint de upload (`agente/router.py`)

```python
if Settings.REDIS_URL:
    task = ingerir_documento_task.delay(agente_id, file.filename, content_b64)
    return JSONResponse(status_code=202, content={"task_id": task.id, ...})
# fallback síncrono
```

Retorna `202 Accepted` + `task_id` quando Celery está disponível.
